# TaoHtml 工作流

TaoHtml 采用“理解确认 → 设计确认 → 制作交付”的状态流程。用户在确认设计简报前，不应收到已经制作的 HTML。

## 1. 选择入口

- 只有想法
- Word / PDF
- 已有 PPT / HTML

当前标准闭环优先覆盖 Word / PDF。其他入口沿用相同确认门，但不承诺当前 Agent 无法读取的格式能力。

## 2. 确认两个基础选择

只在用户没有说明时询问：

- 阅读模式，还是现场演讲模式
- 精简、标准，还是详细

预计页数根据真实材料动态计算，不使用固定模板页数。

## 3. 输出材料理解摘要

完整检查材料后，输出：

- 主题与目的
- 原始结构
- 必须保留的核心观点
- 已有证据和数据
- 发现的缺口、冲突或口径问题
- TaoHtml 对材料逻辑的整体理解

这个阶段不设计页面。用户确认或修正摘要后，才能继续。

## 4. 补齐设计信息

TaoHtml 在内部记录：

```text
已知｜已确认｜自动推断｜仍缺失
```

每次只询问一个会改变结构、证据、视觉或交付结果的问题。不重复询问材料和对话中已经存在的答案；用户授权自动判断后，不换一种说法继续询问。

达到以下状态后立即停止提问：

- 报告目标和受众清楚
- 核心观点完整
- 必要证据已经满足或解决
- 没有会推翻结论的未解决冲突
- 章节结构已选择
- 视觉方向已选择或授权 TaoHtml 决定
- 模式、长度和交付约束已知

## 5. 确认设计简报

设计简报根据项目动态包含：

- 报告目标与受众
- 必须保留的核心观点
- 章节结构
- 每章观点、证据和预期结论
- 视觉与动效方向
- 来源记录
- 交付约束
- 所有自动推断、补全和数据修正

用户针对当前简报回复“确认”后，TaoHtml 才获得制作授权。之前讨论过程中的“同意”“继续”不算设计简报确认。

## 6. 制作 HTML

确认后直接进入内容表达、视觉、页面和 Runtime 制作，不额外生成一份完整文字稿。

当前标准 Runtime 支持：

- 分页阅读与单屏演讲
- 阅读/演讲模式切换
- 动效步进与整页翻页分离
- 返回页面后恢复动效状态
- 全屏、页码和自动隐藏控制

TaoHtml 可以优化结构和表达，但不得遗漏或改变确认过的核心观点。

## 7. QA 与交付

至少运行：

```bash
python skill/taohtml/scripts/check_assets.py path/to/index.html --strict-offline
python skill/taohtml/scripts/check_html_deck.py path/to/index.html path/to/qa
python skill/taohtml/scripts/build_contact_sheet.py path/to/qa path/to/qa/contact-sheet.png
```

QA 需要检查阅读状态、演讲初始状态、逐步前进与后退、整页跳转、返回状态、hash、素材、控制台错误和页面边界。

优先交付单 HTML；素材、字体或附件使单文件明显变慢时，交付 `index.html + assets` 并打包 ZIP。
