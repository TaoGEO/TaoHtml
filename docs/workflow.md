# TaoHtml 工作流

TaoHtml 采用“理解确认 → 视觉解析/确认 → 主题绑定/编译 → 设计确认 → 制作交付”的状态流程。静态参考路线先确认统一 VI 规范图，再把当前 VI 编译成项目专用主题；历史企业档案路线验证并绑定 active version。新建和发生含义变化的续作必须确认当前设计简报，之后才可进入正式制作；交付后不改变含义与结构的局部续作不重跑简报，只读复审不进入生产状态。

## 0. 先识别任务意图，不增加内容入口

TaoHtml 先把本次任务记录为“新建”“只读复审”或“接管续作”。这是覆盖在内容入口之上的任务意图，不是第四个入口；报告内容仍只来自“只有想法 / Word 或 PDF / 已有 PPT 或 HTML”。不能按交接文件名、操作系统路径、公司名或固定关键词判断意图，只看用户要求执行的动作。

- **只读复审**：默认 0 个澄清问题。只读取本次已绑定材料，输出当前状态、来源边界、可续作范围与未核实项；不重启完整问诊，不生成材料理解摘要、设计简报或 HTML。
- **接管续作**：先判断本次变化是否改变含义。纯排版、配色、技术、便携性、动效和不改变含义的局部表达不重跑 intake、材料摘要或设计简报确认，只对精确当前成品执行适用的 QA 与交付核验。修改来源解释、真实数据、证据关系、核心观点、结构、范围承诺或责任边界时，才重建受影响字段、展示并确认完整当前简报；如果确有一个缺口会改变结果，只问最大的那一个，并继续遵守同一套六问上限。

交接中的每个项目必须分别记录来源角色与当前可用状态：

| 来源角色 | 含义 |
|---|---|
| 原始客户材料 | 客户一手文档、真实数据、截图、引语或证据 |
| 外部公开证据 | 当前任务从第三方或公开来源取回的证据；必须记录精确定位、检查覆盖和核验结果，不能标成客户原始材料或 Agent 生成 |
| 二手交接摘要 | Agent 或人员对早期工作的转述，只能用于定位，不能替代原始材料 |
| 当前成品 | 当前 HTML、资源包或导出物，只能证明现在呈现了什么 |
| 视觉参考 | 用于视觉或企业模板解析的已绑定参考 |
| Agent 生成素材 | 补全文案、模拟数据、示意图或占位素材 |
| 仅被描述但当前不可用的材料 | 交接中出现、当前无法检查的材料 |

可用状态固定区分为：`workspace_readable`（工作区可读）、`external_retrieved_inspected`（当前会话或连接器已从外部取回并检查，但不冒充工作区文件）、`platform_visible_not_retrieved`（平台可见但未取回）、`handoff_record_only`（只在交接记录中出现）、`confirmed_missing`（已由用户或权威平台/来源确认丢失）和 `not_yet_verified`（尚未核验）。另行记录 `evidence_verification = verified / unverified / conflicting / not_applicable`；“可读/已取回”只表示可访问，不等于事实已核实。本地命令在一个位置没有找到文件，只能说明该位置未命中，不能写成“已经清理”“确定删除”或“永久丢失”。

接管时，排版、技术、便携性和不改变含义的局部表达可以继续且无需新简报，但必须保留精确交付基线并对修改后成品执行适用检查。修改真实数据、来源归属、证据与观点的关系、真实身份/成果、核心结论、结构、范围承诺或责任边界前，必须恢复原始来源，或取得对该项精确修改的明确确认，并进入含义变化分支；法律、医疗、财务、安全等高风险事实仍需独立核验。普通场景、示例、推演数字、观点补充和表达缺口继续遵循 output-first，并在交付时进入《待核实内容清单》，不因所有原始材料未恢复而一律阻塞。

## 1. 选择入口

- 只有想法
- Word / PDF：选择后请上传源文件，或提供可解析的完整文件路径
- 已有 PPT / HTML：选择后请上传现有文件，或提供可解析的完整文件路径

三个入口在新建或含义变化续作时都必须经过设计简报确认门，但来源理解方式按入口区分；不承诺当前 Agent 无法读取的格式能力。无含义变化的局部续作不重新进入该门。

每次进入新建或含义变化续作时先只建立本次入口。如果当前消息已经给出明确主题，TaoHtml 进入只有想法路线；用户明确选择 Word/PDF 或已有 PPT/HTML 时，先只记录文件入口。如果已有当前会话可用的上传文件、用户提供且可解析的精确路径，或当前任务说明明确绑定的可用来源，可同时建立文件入口与来源绑定，不重复索取。只有入口也未明确时，才显示上面的三个入口，不读取材料、不生成材料理解、设计简报或 HTML。无含义变化的局部续作不重新询问入口；只读复审也不询问入口，有依据时记录已有入口，没有依据时保持未核验。短回答和序号只绑定当前对话中 Agent 最近一次仍有效的选项，文档编号不是跨轮或跨平台命令。规则根据消息是否实际绑定了主题/材料判断，不维护“无效词”清单。

文件路线中，“已选入口”不等于“已绑定来源”。用户只回复 `2` 或 `3`、只说文件类型，或尚未给出可用附件/精确路径时，TaoHtml 必须只请求上传文件或提供可解析的完整路径，然后停止。此时不询问阅读/演示模式、长度、受众或其他后续决策，不搜索本机或工作区，也不开始材料理解。收到已绑定且可访问的文件后才继续，并不重复索取。

本地文件只有三种绑定方式才能进入本次任务：用户当前上传且本会话可用，或用户提供且可解析的精确文件路径；当前任务说明明确声明为输入且当前可访问；用户明确要求“帮我找”或表达等价意思后，Agent 在用户指定或明确纳入范围的窄目录中只做元数据候选发现，展示精确路径并得到用户确认。仅仅存在、处于当前工作目录或工作区、命名为 `input/prompt.md`、位于约定目录或来自历史任务，都不能自动绑定。当前任务授权下由 Agent 从网络或连接器取回的第三方/公开证据使用独立的 `agent_retrieved_external` 绑定，记录精确 URL/稳定定位、取回时间、检查覆盖、支撑主张和核验结果；它绝不能用于绕过本地候选确认。每个实际使用的材料都要在适用的来源账本与设计简报中记录 `source_binding`、来源角色、可用状态、证据核验、检查覆盖与来源理由，且不能把客户材料或公开证据误标为创作性补全。

候选发现默认禁止；选择文件入口、工作区存在或当前 `cwd` 都不是搜索授权。只有用户明确要求 TaoHtml 寻找文件时，才能查看用户指定或已明确纳入范围的窄目录元数据，并先展示精确路径等待确认；确认前不能打开、提取、解析或摘要内容，即使只找到一个候选也不能自动绑定。不能递归扫描用户主目录、桌面、下载目录、平台缓存、云盘根目录或无关工作区来猜测来源。平台已经显示但尚未取回的文件，必须记录为“平台可见未取回”，不能改用广泛文件系统扫描代替平台取回。

## 2. 确认两个基础选择

只有想法路线在入口已明确后可直接进入本步；Word/PDF 与已有 PPT/HTML 必须先完成来源绑定。只在用户没有说明时询问：

- 阅读模式，还是现场演讲模式
- 精简、标准，还是详细

预计页数根据真实材料动态计算，不使用固定模板页数。

## 3. 按入口完成来源理解

只有想法时，直接从当前对话建立「已知｜已确认｜自动推断｜仍缺失」账本，不虚构材料理解摘要。

已有 PPT / HTML 时，检查现有成品并只在忠实迁移和重组都合理时让用户选择。

Word / PDF 时，先运行相应环境预检；PDF 必须在打开或提取前通过 `pdf` profile。之后完整检查已绑定材料并输出材料理解摘要：

- 主题与目的
- 原始结构
- 必须保留的核心观点
- 已有证据和数据
- 发现的缺口、冲突或口径问题
- TaoHtml 对材料逻辑的整体理解

这个阶段不设计页面。Word / PDF 的摘要经用户确认或修正后，才能继续；确认过的真实来源和数据此后不得被改写成创作性补全。

含义变化的续作只补充受本次修改影响的材料理解，并保留“原始材料 / 外部公开证据 / 二手摘要 / 当前成品 / 视觉参考 / Agent 生成 / 当前不可用”的差异；二手摘要再详细，也不能被说成足以支撑所有后续修改。无含义变化的局部续作不重建或重新确认材料摘要，只读复审也不为交接审计虚构一份材料理解摘要。

## 4. 补齐设计信息

TaoHtml 在内部记录：

```text
已知｜已确认｜自动推断｜仍缺失
```

每次只询问当前最大且会改变报告设计结果的缺口。不重复询问材料和对话中已经存在的答案；用户授权自动判断后，不换一种说法继续询问。清楚输入可以 0 问进入简报；普通问诊通常 3–5 问，Agent 主动澄清最多 6 问。

如果目标要求受众试用、购买、预约、下载、安装、联系或注册，需要把「期望行动」与「真实执行路径」分开记录。真实路径必须来自用户、来源材料或项目上下文，或由用户授权 TaoHtml 选择后经 Agent 验证；已有可核验入口时不重复询问。不得编造网址、二维码、联系方式、价格、命令或产品入口。知识说明、教育内容和内部汇报若不要求外部行动，不额外询问行动渠道。

缺少真实路径会让转化目标无法完成时，它是最小硬边界：只问一个最小问题，并纳入同一套 6 问上限。达到停止条件仍无法确认时，暂停问诊；只有用户明确同意把目标降级为不要求直接行动，才能继续进入简报。

普通信息不足不触发无止境追问。缺少的场景、数字、观点、示例和表达可以作为创作性补全进入制作，并在交付时逐条披露。只有真实客户/公司身份、引语、文献或来源，已经发生的客户成果，高风险法律/医疗/财务/安全事实，以及已确认的真实来源、数据和行动渠道等最小硬边界不能推断或替换。

达到以下状态后立即停止提问：

- 报告目标和受众清楚
- 核心观点完整
- 必要真实证据已经满足、明确边界，或与计划中的创作性补全清楚分离
- 没有会推翻结论的未解决冲突
- 章节结构已选择
- 模式、长度和交付约束已知

## 4A. 内容清楚后进入独立设计选择门禁

普通澄清结束后，TaoHtml 才进入设计选择。内置主题和动效密度不计入普通 6 问，因此即使已经问满 6 次，或连续 3 轮没有获得有效新信息，也不能自动跳过。它们也不会因为模型判断“风险较低”而被代选。

没有可复用企业档案、没有客户参考、客户也没有明确指定主题时，TaoHtml 会一次展示当前完整内置目录：黑白荧光卡片、严谨咨询报告、稳重企业年报、杂志图文拼贴。每项都带精确名称、一句话画面说明和 preview；可以标记 1–2 个“更推荐”，但不会隐藏其余选项。只有客户主动给出了能明确映射到既有类别的范围约束时才展示对应子集。客户已明确点名具体主题时直接记录，不重复展示目录；否则必须等待客户选择，或等待客户明确说“由 TaoHtml 决定”。只有后一种明确委托才允许 TaoHtml 按推荐匹配选择最低风险项并记录委托引用。

动效密度使用三个原生值：`minimal`（少量）、`moderate`（适中）、`rich`（丰富）。TaoHtml 可以推荐一个，但客户仍需选择或明确委托；静态参考图、企业档案和 Workflow Profile 默认值都不能替代该决定。主题和动效可以在同一轮展示，但各自记录“客户选择 / 明确委托”、具体值和当前会话引用。已经记录的选择不重复询问。

静态参考与企业档案继续走各自 VI / binding / 项目主题门禁，不额外要求选择内置主题；它们只需完成独立的动效决定。上述选择未完成时不能生成或确认设计简报，更不能进入正式 HTML。

## 5. 解析企业模板档案

内容和结构清楚后，TaoHtml 先从本次合格材料与对话解析企业身份，再到 `TAOHTML_HOME`（默认 `~/.taohtml`）做 display name、profile id 和明确别名的精确匹配。它不增加固定启动问卷，也不把上一次项目的选择升级成永久用户偏好。

唯一 active profile 匹配时，TaoHtml 自动为当前任务生成 `profile-use` binding，记录 profile id、version、theme fingerprint、VI/参考图 hashes、解析依据和临时 override 状态。面向客户只说明“本次沿用【企业模板 vN】；如需更换请直接说明”，不重新读取参考图、不重新生成 VI，也不询问“是否复用”。多个候选、企业身份不明确、别名冲突或当前任务模式与档案冲突时只问一个选择问题；禁止猜测和跨企业混用。

三种更换语义严格分离：本次更换只产生临时 override；以后更新必须重新走企业 VI 确认并创建不可变 v2/v3，再原子切换 active pointer；另一家公司创建或选择独立档案。旧版本保留回退，v1 只提供归档、不做默认破坏性删除。

档案不写入 Skill 安装目录，也不保存报告正文、项目目标、受众、证据或客户报告数据。它只保留企业模板所需的 VI 合同、参考图、经现有 loader 验证的 theme、确认边界和完整文件 hashes。本机不同 Agent 可以复用同一 TaoHtml home；跨设备或无持久磁盘环境必须显式 export/import 完整包，不宣称自动云同步。

档案 binding 只替代本次重复 VI 环节。当任务进入新建或含义变化的简报门禁时，当前《报告设计简报》仍需确认，production authorization、正式 HTML、浏览器 QA 和交付门禁都不能绕过。无含义变化的局部续作不因档案复用而重开该门禁，但仍对精确成品执行适用检查。active version、企业身份、VI、参考图、theme 或 manifest 任一漂移都使 binding 失效，不能静默回退内置主题。

## 6. 静态参考的双模式 VI 确认

用户选择“使用我的参考”时，TaoHtml 先确定参考图模式。参考风格重构接受 1 张 PNG/JPEG/WebP 静态图；企业模板保真接受同一模板族 1–3 张代表性静态图。用户已经明确说“企业模板保真”“公司模板原样采用”等，不重复询问；意图不清楚时只问一次二选一，并计入原有 6 问上限：

- 参考风格重构：提取设计语言，允许重新构图和创新。
- 企业模板保真：锁定截图中可见的 Logo、页眉、页脚、品牌条和固定装饰，只设计安全内容区。

打开或分析任何参考图前，必须运行 `static-reference` profile。它检查 Pillow、PyYAML、Python Playwright，并真实启动 Chromium 生成最小截图；子进程崩溃和超时会快速失败。两种参考模式共用这条依赖链，因此失败时不能手工绕过企业保真，也不能降级为同样依赖该链的参考风格重构。用户只能选择修复/更换环境后重试，或明确放弃客户参考路线并改用四套内置视觉系统。

两种模式继续共用同一条“静态参考图 → VI 标准图 → 当前产物确认 → 项目主题编译”流程。TaoHtml 先检查当前会话能否可靠定位标题、色块、边框等可见事实。可读时，只分析静态设计语言，并通过确定性模板输出一张 3200×2400 的统一《VI 设计标准图》。重构模式延续原有构图、层级和组件确认；企业保真还必须自动识别每张来源图的 cover/toc/section/content/data 角色（真歧义才追问），并展示：

- 1–3 张来源缩略图、角色、canvas_bbox，先裁截图边框再归一化
- 每个 observed shell 的固定区域覆盖层、逐项裁切预览和可编辑安全区
- 可延展设计语言，以及 cover/toc/section/content/data 五个页面壳小样
- unknown 限制，包括无法从截图恢复的母版、矢量 Logo、字体源文件或不可见资产

三类边界是“直接观察｜报告适配建议｜参考中无法判断”。任何未在来源图中出现的页面角色只能标为 `extension/proposed`，不得伪装成 `observed`。企业保真承诺“截图中可见效果保真”，不承诺恢复原始 PPT 母版、矢量 Logo 或字体源文件；禁止 AI 重绘 Logo，优先从最清晰截图裁切，所有截图都不足时要求更清晰截图。本阶段不宣称支持独立 Logo 上传。用户可在 VI 确认前纠正角色、锁定元素和安全区；当前产物得到明确确认并记录会话引用后，当前 JSON 与参考图才能进入项目主题编译，生成过程不得擅自改变。该确认不等于设计简报确认，也不授权正式报告制作。

PPT、网页、视频、状态序列、超过 3 张企业截图或多张重构参考不进入该流程；TaoHtml 会要求支持的代表性静态输入，不会把它当成“无参考”而自动推荐内置风格，也不会推断动效。PNG/JPEG/WebP 必须是单帧；企业截图裁边后的画布必须在 0.25% 相对容差内符合 16:9，超出时拒绝而不拉伸。

若当前会话无法可靠读取参考图，TaoHtml 明确停止，让用户选择手动调整平台会话后重开，或改用四套内置视觉系统。Skill 不维护具体模型清单，也不假设任务中能自动切换模型。

## 7. 编译项目专用主题

只在当前 VI 产物得到明确确认后执行。TaoHtml 创建一份最小 handoff JSON，记录项目 id 与名称、确认状态与当前会话引用、VI JSON 和有序参考图的相对路径及 SHA-256、目标阅读/演讲模式、已并入当前 VI 的客户修正。路径、顺序、schema、文件或哈希任何一项不匹配，编译都必须在写输出前失败。

确定性编译器输出当前项目专用的 `theme.json`、`theme.css`、`templates.html` 和 `provenance.json`。它直接从可执行布局语法选择不同的 DOM 分支、结构 CSS、layout variants 与 composition。企业保真模式会从裁后画布精确裁切共享固定资产、离线嵌入并记录来源页/源图/裁切哈希；封面、目录、章节、普通内容、数据页按角色选择不同 shell。固定层不得移动或参与动效，所有正文、排版和 Runtime 分步呈现只进入对应可编辑区。整张含示例正文的截图不会成为页面背景。

`observed` 和 `extension` 只是 eligible；只有实际进入 token、CSS、模板、manifest 执行字段或明确 guardrail 的记录才可写 `compiled: true`，并必须列出具体 usage targets。未使用项保持 `compiled: false`；`unknown` 始终不编译。运行所需而 VI 未提供的值只能使用中性可逆 fallback，并在 provenance 中单独记录，不能升级为参考事实。动效沿用 TaoHtml Runtime 与正式报告任务的通用 `fragment` / `data-step` 语法，manifest 明确标记它不是从静态参考观察或推断得到。

项目主题不是第五套内置风格，不进入内置主题目录或选择表。生产渲染器用 `--project-theme` 显式加载它，同时继续复用同一 Runtime、翻页、返回状态、全屏和来源标签合同。

企业模板保真主题只有在当前 VI 已确认、handoff hashes 完整、当前项目 theme 已编译并通过既有 project-theme loader 后，才能创建企业档案 v1。档案会用企业 profile 身份重新编译一份不含原项目名称/修正的等价 theme，并与已验证来源 theme 比较结构、CSS 和模板；保存后仍由同一个 loader 验证。永久更新重复这条链路创建新版本，临时 override 不创建版本。

## 8. 确认设计简报

本节只适用于新建或含义变化续作。无含义变化的局部续作不生成或重新确认设计简报；它直接以精确已交付成品为语义基线，修改后运行适用 QA 与交付验证。

设计简报根据项目动态包含：

- 报告目标与受众
- 期望行动；转化型报告还需记录真实执行路径、渠道来源与验证状态、最终页面展示方式
- 必须保留的核心观点
- 章节结构
- 每章观点、证据和预期结论
- 视觉与动效方向
- 来源记录
- 交付约束
- 所有会改变设计结果的自动推断、预计创作性补全范围和真实数据修正

在本节适用时，用户针对当前简报给出明确确认，且 TaoHtml 记录该产物与当前会话引用后，才获得制作授权。之前讨论过程中的模糊赞同或继续指令不算当前设计简报确认。

新建或含义变化续作还要维护当前任务的机器可检查授权状态：Word/PDF/PPT/HTML 材料路线的当前材料理解、参考路线的当前 VI 或历史档案路线的当前 binding、项目主题编译/验证、适用的内置主题决定、所有路线的动效密度决定和当前设计简报按适用性依次完成。主题与动效分别记录具体值、`user_selected | delegated_to_taohtml` 和当前会话引用；任何 `pending` 都会阻止简报和正式动作。对话中的摘要/简报在进入确认门前必须保存为当前任务根目录下的客户可读产物，确认状态绑定安全相对路径和当前字节的 SHA-256；profile binding 同样绑定任务本地路径/hash，并实时验证 TaoHtml home。材料摘要、VI 标准图、profile binding、设计选择提示和设计简报只能在对应状态生成/更新；正式 HTML、浏览器 QA 与交付只有在 `check_production_authorization.py --artifact-root <当前任务根>` 对相应 action 返回允许时才能执行。旧的 `1.1/1.2` production state 缺少这两项决定，只能以阻塞状态读取；必须迁移到 `1.3`、补齐选择证据、更新并重新确认完整简报，不能沿用旧确认静默放行。确认后文件变化，或档案 active version/identity/theme/VI/reference、记录的设计决定变化，都会使门禁失效。不得把近似成品的 HTML 叫作“确认预览”绕过门禁。无含义变化的局部续作不伪造新的授权状态或强制重新确认简报，但必须对精确基线和修改后成品执行适用的素材、浏览器、Runtime/编辑器、语义追踪与交付核验。

## 9. 制作 HTML

新建或含义变化续作在当前简报确认后，直接进入内容表达、视觉、页面和 Runtime 制作，不额外生成一份完整文字稿。无含义变化的局部续作则从精确交付成品直接进入受限修改和适用验证，不重开问诊、材料摘要或设计简报确认。

当前标准 Runtime 支持：

- 分页阅读与单屏演讲
- 阅读/演讲模式切换
- 动效步进与整页翻页分离
- 返回页面后恢复动效状态
- 全屏、页码和自动隐藏控制
- “更多 → 编辑模式”内直接修订报告文字、替换图片和拖动裁切焦点
- 统一撤销/重做、标签页会话级刷新恢复，以及不覆盖原文件的新 HTML 导出

TaoHtml 可以优化结构和表达，但不得遗漏或改变确认过的核心观点。

## 10. QA 与交付

至少运行：

```bash
python skill/taohtml/scripts/check_assets.py path/to/index.html --strict-offline
python skill/taohtml/scripts/preflight.py --profile browser --workspace /path/to/workspace
python skill/taohtml/scripts/check_html_deck.py path/to/index.html path/to/qa
python skill/taohtml/scripts/check_editor_runtime.py path/to/index.html path/to/editor-qa
python skill/taohtml/scripts/build_contact_sheet.py path/to/qa path/to/qa/contact-sheet.png
```

浏览器预检通过后才能开始浏览器 QA；失败或未运行不能写成“浏览器 QA 通过”。QA 需要检查阅读状态、演讲初始状态、逐步前进与后退、整页跳转、返回状态、hash、素材、控制台错误和页面边界。初始为演讲模式的整份报告必须至少有一个 `fragment-v1` 受控讲解步骤；阅读模式可以为零。每页 active slide 的实际渲染矩形必须在小容差内覆盖 deck/canvas。每页需要检查的呈现状态还要检测独立可见 HTML 文字与 SVG `<text>` 的真实矩形碰撞，并保留 1px 安全间距。SVG 文字、绝对定位/自身变换的 HTML 文字以及布局盒本身相交的 HTML 文字保持严格失败；只有两个静态、无自身变换的 HTML 正常流布局盒已分离，且 Range 仅有不超过 1.25px 的浅层字体度量交叠时，才记为可审计的度量排除而非失败。只有具体文字 owner 上带理由的 `data-qa-ignore-text-collision` 才能局部豁免，所有豁免和正常流度量排除都会进入 QA JSON。转化型报告还要从简报追踪到最终页面，确认真实行动入口可见、可用、与目标一致；链接、命令、联系方式、价格和二维码必须与已验证来源一致，不能只检查 CTA 文案里是否有动词。

当前成品未通过适用的 production authorization、严格离线素材检查、browser preflight、当前 HTML 浏览器 QA、简报到成品追踪和交付核验前，只能说“已找到”“可以预览”或“尚未核验”，不能说“已就绪”“QA 已通过”或“可正式交付”。操作说明必须来自对当前 HTML 的实际测试；如果只依据当前 Runtime 合同描述通用行为，必须明确说明尚未核验该交接成品是否实现了这项行为，不能照抄二手交接摘要中的控制说明。

优先交付单 HTML；素材、字体或附件使单文件明显变慢时，由 Agent 交付 `index.html + assets` 并使用仓库脚本打包 ZIP。浏览器内编辑器本身只导出新 HTML：含相对 assets 时必须保留原目录结构，不宣称浏览器会生成 ZIP。成品完成后自动附《待核实内容清单》，逐条给出页面/内容、补充类型、来源状态和建议动作。普通推演集中在交付说明；模拟图表、虚构案例和容易被误认成真实证据的数据还必须在 HTML 相邻位置标“示意 / 模拟 / 待核实”。
