# TaoHtml 工作流

TaoHtml 采用“理解确认 → 视觉确认 → 设计确认 → 制作交付”的状态流程。单张静态参考路线先确认 VI 规范图；所有路线都必须确认设计简报，之后才可制作 HTML。

## 1. 选择入口

- 只有想法
- Word / PDF
- 已有 PPT / HTML

三个入口都必须经过设计简报确认门，但来源理解方式按入口区分；不承诺当前 Agent 无法读取的格式能力。

## 2. 确认两个基础选择

只在用户没有说明时询问：

- 阅读模式，还是现场演讲模式
- 精简、标准，还是详细

预计页数根据真实材料动态计算，不使用固定模板页数。

## 3. 按入口完成来源理解

只有想法时，直接从当前对话建立「已知｜已确认｜自动推断｜仍缺失」账本，不虚构材料理解摘要。

已有 PPT / HTML 时，检查现有成品并只在忠实迁移和重组都合理时让用户选择。

Word / PDF 时，完整检查材料后输出材料理解摘要：

- 主题与目的
- 原始结构
- 必须保留的核心观点
- 已有证据和数据
- 发现的缺口、冲突或口径问题
- TaoHtml 对材料逻辑的整体理解

这个阶段不设计页面。Word / PDF 的摘要经用户确认或修正后，才能继续；确认过的真实来源和数据此后不得被改写成创作性补全。

## 4. 补齐设计信息

TaoHtml 在内部记录：

```text
已知｜已确认｜自动推断｜仍缺失
```

每次只询问当前最大且会改变报告设计结果的缺口。不重复询问材料和对话中已经存在的答案；用户授权自动判断后，不换一种说法继续询问。清楚输入可以 0 问进入简报；普通问诊通常 3–5 问，Agent 主动澄清最多 6 问。

如果目标要求受众试用、购买、预约、下载、安装、联系或注册，需要把「期望行动」与「真实执行路径」分开记录。真实路径必须来自用户、来源材料或项目上下文，或由用户授权 TaoHtml 选择后经 Agent 验证；已有可核验入口时不重复询问。不得编造网址、二维码、联系方式、价格、命令或产品入口。知识说明、教育内容和内部汇报若不要求外部行动，不额外询问行动渠道。

缺少真实路径会让转化目标无法完成时，它是最小硬边界：只问一个最小问题，并纳入同一套 6 问上限。达到停止条件仍无法确认时，暂停问诊；只有用户明确同意把目标降级为不要求直接行动，才能继续进入简报。

普通信息不足不触发无止境追问。缺少的场景、数字、观点、示例和表达可以作为创作性补全进入制作，并在交付时逐条披露。只有真实客户/公司身份、引语、文献或来源，已经发生的客户成果，高风险法律/医疗/财务/安全事实，以及已确认的真实来源、数据和行动渠道等最小硬边界不能推断或替换。

达到以下状态后立即停止提问：

- 报告目标和受众清楚
- 核心观点完整
- 必要真实证据已经满足、明确边界，或与计划中的创作性补全清楚分离
- 没有会推翻结论的未解决冲突
- 章节结构已选择
- 视觉方向已选择或授权 TaoHtml 决定
- 模式、长度和交付约束已知

## 5. 单张静态参考的 VI 确认

用户选择“使用我的参考”并只提供一张静态图时，TaoHtml 先检查当前会话能否可靠定位标题、色块、边框等可见事实。可读时，只分析静态设计语言，并通过确定性模板输出一张 3200×2400 的《VI 设计标准图》，至少展示：

- 参考缩略图、颜色与色值、字体层级
- 网格、留白、间距和组件语言
- 图片处理以及参考中实际出现的图表/证据语言
- 封面、内容页、数据页微缩示例
- 保留项、禁用项和三类边界标签

三类边界是“直接观察｜报告适配建议｜参考中无法判断”。参考没有展示的类别保持未知，不得补写为观察事实。用户回复“确认 VI”后，当前 JSON、HTML、PNG 和参考图才可交给下一独立任务生成项目专用主题；该确认不等于设计简报确认，也不授权正式报告制作。

只有恰好一张静态参考图进入该流程。明确参考若为 PPT、网页、视频、多张图或状态序列，v1 会说明暂不支持并请用户提供一张代表性静态截图；不会把它当成“无参考”而自动推荐内置风格，也不会推断其中的动效。

若当前会话无法可靠读取参考图，TaoHtml 明确停止，让用户选择手动调整平台会话后重开，或改用四套内置视觉系统。Skill 不维护具体模型清单，也不假设任务中能自动切换模型。

## 6. 确认设计简报

设计简报根据项目动态包含：

- 报告目标与受众
- 期望行动；转化型报告还需记录真实执行路径、渠道来源与验证状态、最终页面展示方式
- 必须保留的核心观点
- 章节结构
- 每章观点、证据和预期结论
- 视觉与动效方向
- 来源记录
- 交付约束
- 所有会改变设计结果的自动推断、预计创作性补全范围和真实数据修正

用户针对当前简报回复“确认”后，TaoHtml 才获得制作授权。之前讨论过程中的“同意”“继续”不算设计简报确认。

## 7. 制作 HTML

确认后直接进入内容表达、视觉、页面和 Runtime 制作，不额外生成一份完整文字稿。

当前标准 Runtime 支持：

- 分页阅读与单屏演讲
- 阅读/演讲模式切换
- 动效步进与整页翻页分离
- 返回页面后恢复动效状态
- 全屏、页码和自动隐藏控制

TaoHtml 可以优化结构和表达，但不得遗漏或改变确认过的核心观点。

## 8. QA 与交付

至少运行：

```bash
python skill/taohtml/scripts/check_assets.py path/to/index.html --strict-offline
python skill/taohtml/scripts/check_html_deck.py path/to/index.html path/to/qa
python skill/taohtml/scripts/build_contact_sheet.py path/to/qa path/to/qa/contact-sheet.png
```

QA 需要检查阅读状态、演讲初始状态、逐步前进与后退、整页跳转、返回状态、hash、素材、控制台错误和页面边界。转化型报告还要从简报追踪到最终页面，确认真实行动入口可见、可用、与目标一致；链接、命令、联系方式、价格和二维码必须与已验证来源一致，不能只检查 CTA 文案里是否有动词。

优先交付单 HTML；素材、字体或附件使单文件明显变慢时，交付 `index.html + assets` 并打包 ZIP。成品完成后自动附《待核实内容清单》，逐条给出页面/内容、补充类型、来源状态和建议动作。普通推演集中在交付说明；模拟图表、虚构案例和容易被误认成真实证据的数据还必须在 HTML 相邻位置标“示意 / 模拟 / 待核实”。
