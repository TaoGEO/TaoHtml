# Report IR v0 对照实验设计简报｜Case B

## 项目目标

把“客户访谈很多，但洞察没有进入产品决策”制作成一份可直接现场汇报的 16:9 离线 HTML 演示。

## 受众与行动

- 受众：正在建立用户研究和产品决策机制的产品团队。
- 期望理解：团队缺的不是更多访谈，而是让访谈证据进入需求判断、方案取舍和上线复盘的闭环。
- 期望行动：先跑通一个需求闭环，再扩大研究范围。

## 使用模式

- 现场演讲。
- 5 页。
- 内容按照讲解顺序逐步出现。
- 最终阅读状态必须显示完整内容。
- 完全离线运行。

## 视觉系统

- 主要主题：黑白荧光卡片。
- 需要能在不修改内容的情况下重新编译为严谨咨询报告。

## 固定页面内容

### 第 1 页：核心命题

- 小标题：`01 / 核心命题`
- 主标题第一行：`洞察入链，`
- 主标题第二行：`产品决策才可复盘`
- 导语：`把散落的客户访谈变成可追踪、可比较、可进入决策的洞察系统。`
- 标签：`核心判断`
- 结论：`团队缺的不是更多访谈，而是一套让客户证据进入需求判断、方案取舍与上线复盘的结构。`

### 第 2 页：当前断点

- 小标题：`02 / 当前断点`
- 标题：`访谈不少，决策依据依然断层`
- 导语：`同一批声音散落在录音、纪要和个人印象里，最后只剩结论，没有可复查的判断过程。`
- 断点 01：`问题失焦`；说明：`访谈问题与产品决策没有绑定，记录很多却无法回答当前取舍。`
- 断点 02：`证据失真`；说明：`高频意见被当成普遍事实，反例与适用边界没有保留。`
- 断点 03：`决策失联`；说明：`需求上线后无法追溯当初采用了哪条洞察，也无法判断是否有效。`

### 第 3 页：系统机制

- 小标题：`03 / 系统机制`
- 标题：`四步把访谈变成可追踪洞察`
- 导语：`每一步都保留来源、判断和下一步决策入口。`
- 01 设题：`先绑定要做的产品决策与需要验证的问题。`
- 02 切片：`拆出原话、场景、痛点、行为和反例。`
- 03 聚合：`按问题归并证据，并标注强度与适用边界。`
- 04 回写：`把洞察绑定到需求、方案取舍和上线验证。`

### 第 4 页：固定合成样本

- 小标题：`04 / 固定样本`
- 标题：`同一批访谈，直接看洞察转化`
- 导语：`这是用于架构对照的合成样本，不代表真实客户研究数据。`
- 来源标签：`来源：本地合成访谈记录；示意 / 待核实`
- 指标 1：`15`；名称：`已完成访谈`
- 指标 2：`10`；名称：`已形成洞察`
- 指标 3：`7`；名称：`已绑定需求`
- 表头：`阶段 / 数量 / 判断`
- 访谈完成 / 15 / `样本记录已统一`
- 洞察形成 / 10 / `仍需补齐反例与边界`
- 需求绑定 / 7 / `优先验证决策是否采用`

这些数字只来自 `input/materials/evidence.svg`，只用于合成实验，不能写成真实客户研究成果。

### 第 5 页：下一步

- 小标题：`05 / 下一步`
- 标题：`先跑通一个需求闭环，再扩大研究范围`
- 导语：`用三周跟踪一个真实需求，让客户声音从访谈进入决策与复盘。`
- 01 选需求：`锁定一个正在判断优先级的核心需求。`
- 02 补链路：`为 10 条关键洞察补齐原话、反例与适用边界。`
- 03 验闭环：`记录洞察采用、方案变化和上线验证结果。`
- 标签：`行动结论`
- 结论：`只有能回到产品决策并被结果检验的洞察，才是可复用的研究资产。`

## 证据与披露边界

- `input/materials/evidence.svg` 是本地合成 fixture，字节身份可核对，但不是现实世界证据。
- 第 4 页数字必须保留“合成样本、示意或待核实”边界。
- 不新增真实客户、企业、引语、价格、网址、邮箱或电话号码。
- 最终交付说明必须包含简洁的《待核实内容清单》。

## 防复用身份

- 本案例 ID：`case-b`。
- Report IR 的 `report.id` 必须是 `report_customer_insight_loop_b`。
- 不得复制其他案例的标题、数字、结论或交付文件。

## 交付要求

- 入口：`deliverable/index.html`。
- 浏览器双击即可运行。
- 不依赖 CDN、在线字体、远程脚本或远程图片。
- 支持 TaoHtml 当前翻页、动效、阅读最终状态和全屏 Runtime。
